ListManager.js

An instance-based utility for managing dynamic lists within server-rendered pages.

Essentially, an instance-based implementation of initListItems().

Its primary purpose is to allow users to dynamically add, remove and optionally soft-delete sections of a form while keeping a single server-rendered template as the source of truth.

Motivation

Our original initListItems() utility worked well but was architected as a global behavior system. Elements were linked together via selectors:

Add button -> data-template selector -> template -> data-list selector -> list -> count target selector

This created loose coupling throughout the page where the relationship between elements was challenging to discover without understanding all required data attributes.

Goals of redesign:

  • Instance based
  • Group ownership under a single root element
  • Eliminate cross-element selectors
  • Configurable via defaults (preferred) + options + dataset
  • Maintain support for nested dynamic lists
  • Preserve server-rendered templates as the source of truth
  • Minimal, simple API

Design Philosophy

Decorate a root

Instead of connecting independent DOM elements together, ListManager decorates a single root element.

new ListManager('#widget-list');

Everything inside that root becomes part of the component.

ListManager Root
├── Add button (data-list-add)
├── Remove button (data-list-remove)
├── List count display (data-list-count)
├── Item number display (data-list-item-number)
├── List (data-list) - required
└── Item Template (<template>) - required

Templates remain server rendered

ListManager intentionally does not generate HTML; the server remains the source of truth. A standard <template> contains the markup.

Soft-delete (not DOM removal) if flagged

Persisted records often cannot simply be removed from the DOM. Instead, add data-list-flag-delete to a hidden input to set that value to true when removed from the list.

<input type="hidden" data-list-flag-delete name="widgets[0].deleted">

The server can then process deletions based on this flag being submitted.

Non-persisted items are removed from the DOM entirely.

Ownership boundaries

ListManagers may be nested. A ListManager only owns descendants whose nearest ListManager ancestor is itself.

Single Source of Truth

The examples show only the HTML used in list management. In practice, we'd use a reusable server-side component (e.g., JSP/JTE tag or fragment) to define and render a single item. The item would be rendered into one of two locations based on whether the item is existing or new.

  • Existing items - in the data-list from a forEach looping over items in the view model and passing them into a tag to render.
  • New items - within the <template> via the same tag but without an item (using a placeholder index string, {index}, if needed).

Doing this ensures there is a single source of truth as to how a list item is represented in HTML.

Bottom line: The duplication you see in this example HTML should not be present in the server-side templates used to construct the markup.

Simple List

List<String>

A simple list could be a persisted list of strings annoated with @ElementCollection or primitive values on a DTO list. Because we are dealing with just values, indexes are not needed nor is sending a delete flag to the server.

If all items are removed from the list, however, we need a way to inform the server that the list should be cleared. In Spring MVC, we do that using a field marker prefix (underscore _ by default).

Try submitting an empty list in the example below. The presence of the _names parameter without additional names singals to Spring that the list is empty.

This example shows sending the server a list of strings having the request parameter name names.

Existing items are rendered within the data-list* attributed container element.

New items are defined by the <template> markup.

An item is expected to have a single parent element (i.e., a single item should not contain sibling elements).

*[data-list] is the default list attribute selector.

Indexed List

@OneToMany List<Entity>

An indexed list is often a persisted @OneToMany list on a parent entity. Because items are persisted, they must be explicitly removed server-side using the delete flag usually tracked using an @Transient private boolean deleted; property on the entity itself.

In this example we are not sending just a list of strings to the server, but values associated to an indexed (persisted) item (e.g., serial and title properties on a Widget in this example).

Because existing items are persisted in the database, a hidden input having a data-list-flag-delete attribute is used to flag deletion. This instructs the ListManager to hide the list item and set the data-list-flag-delete input's value to true (a soft-delete). The actual deletion of the item (e.g., delete database record) happens server-side.

It is best to only include data-list-flag-delete on existing, persisted items. Flagging a delete on a new item is unnecessary; just don't submit that item to the server. This is exactly how ListManager behaves when it removes an item not having a data-list-flag-delete input element--that item is removed from the DOM and nothing is sent to the server.

Look at the difference upon submission between removing an existing versus new item from the list.

Widgets

Insertion Location

Item insertion can be either at the top or bottom of the list. This is controlled by the data-list-insertion attribute.

This example also demonstrates overriding default selector values using data attributes. Don't do this unless there is a real need. Otherwise use the defaults.

Ensure indexes are reversed for existing persisted items (i.e., 0 at the bottom and incrementing up). This avoids issues when you reload the page.
Names

Nested Lists

Add/remove events are scoped to their immediate, owning ListManager. Nested lists should be given a unique index placeholder (via data-list-index-placeholder on the root element).

Note: newly added items which themselves contain an element intended to be a ListManager need to be initialized as ListManager instances when added to the DOM. The utils.js method initAll already does this by default registering a MutationObserver to run the init callback on any matching elements added to the DOM.

Contacts

Options

Setting names are prefixed with list to help distinguish them when used as data attributes (their intended primary usage).

<div data-isp-toggle="list-manager"
     data-list-transition-time=".5"
     data-list-index-placeholder="{i}"
     data-list-selector="[data-widgets]"
     data-list-insertion="top"
     data-list-add-selector="[data-add-widget]"
     data-list-remove-selector="[data-remove-widget]"
     data-list-count-selector="[data-widget-count]"
     data-list-item-number-selector="[data-widget-number]"
     data-list-flag-delete-selector="[data-widget-flag-delete]">...

Prefer the defaults in almost all cases.

Name Type Default Description
listTransitionTime number .25 Duration of add/remove animations in seconds.
listIndexPlaceholder string {index} Placeholder text replaced with the next item index when cloning template content.
listSelector string [data-list] Selector used to locate the list container within the root element.
listInsertion 'top'|'bottom' 'bottom' Where newly added items are inserted into the list.
listAddSelector string '[data-list-add]' Selector used to locate add buttons within the root element.
listRemoveSelector string '[data-list-remove]' Selector used to locate remove buttons within the root element.
listCountSelector string '[data-list-count]' Selector used to locate elements that display the total visible item count.
listItemNumberSelector string '[data-list-item-number]' Selector used to locate elements that display an individual item's visible position.
listFlagDeleteSelector string '[data-list-flag-delete]' Selector used to locate hidden inputs that should be set when an item is removed (soft-deleted).

Methods

Method Description Example
